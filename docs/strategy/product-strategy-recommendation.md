## 1. 執行摘要

### 1.1 核心建議

基於對 Labor Law Assistant PRD 的分析，針對您提出的「雙資料來源 RAG 系統」策略，我提出以下核心建議：

#### ✅ 建議採納的策略

1. **雙資料來源定位明確**
   - 法條資料庫 = 權威基礎（Primary Source）
   - 內部文章庫 = 實務指引（Secondary Source）
   - 兩者互補而非競爭

2. **使用者介面清楚區分**
   - 視覺差異化（法條用書本圖示📕、內部文章用文件圖示📝）
   - 分層呈現（直接回答 → 白話說明 → 法條依據 → 實務案例）
   - 明確標示資訊來源與更新日期

3. **MVP 範圍建議**
   - 法規：8 大法規核心條文（約 400 條）
   - 內部文章：Top 100 FAQ + 10 篇行動指南
   - 時程：28 週（約 7 個月）

4. **資料更新機制**
   - 法條：24 小時 SLA（自動監控 + 人工審核）
   - 內部文章：7 天 SLA（用戶回饋驅動）

#### 定位對照表

| 維度 | 法條資料庫 | 內部文章資料庫 |
|------|-----------|---------------|
| **角色** | 系統的「法律憲法」 | 系統的「實務手冊」 |

| **目的** | 提供法律依據 | 提供理解與指引 |
| **語言** | 法律術語 | 白話文 |
| **結構** | 條文式 | 問答式、案例式 |
| **變動性** | 低（修法才變） | 高（持續優化） |
| **完整性** | 法條完整但難懂 | 精選重點且易懂 |
| **使用時機** | 所有回答都需引用 | 補充說明、案例、行動指引 |
| **品質保證** | 官方來源驗證 | 三級人工審核 |
| **更新頻率** | 修法時（低頻） | 用戶回饋（高頻） |

### 2.3 是否需要區分顯示？

#### 建議：必須明確區分，原因如下

**理由 1：建立信任（Trust Building）**
- 用戶需要知道資訊來源的權威性
- 清楚標示「這是法律」vs「這是實務說明」
- 避免混淆導致誤解

**理由 2：法律風險管理（Legal Risk）**
- 內部文章無法律效力，不能與法條混為一談
- 如果出錯，清楚來源可降低責任
- 符合「透明可溯」的產品憲章

**理由 3：用戶體驗優化（UX）**
- 不同用戶有不同需求：
  - 一般勞工：優先看白話文（內部文章）
  - HR 專員：優先看法條原文
  - 小企業主：優先看行動指南
- 清楚區分可讓用戶快速跳到所需資訊

**理由 4：品質控制（Quality Control）**
- 法條來源：官方資料庫（幾乎不會錯）
- 內部文章：人工撰寫（可能有錯）
- 分開標示可讓用戶判斷可信度

## 3. 優先級排序（RICE 框架）

### 3.1 RICE 評分方法

**公式**：RICE Score = (Reach × Impact × Confidence) / Effort

- **Reach（觸及）**：影響多少用戶？（每季預估用戶數）
- **Impact（影響）**：對用戶價值多大？（3=巨大, 2=高, 1=中, 0.5=低, 0.25=極小）
- **Confidence（信心）**：我們有多確定？（100%=高, 80%=中, 50%=低）
- **Effort（工時）**：需要多少人月？（Person-Months）

---

### 4.2 關鍵風險深入分析

#### 風險 R1：AI 錯誤資訊導致用戶損失

**情境範例**
```
用戶：「我工作 3 年被資遣，資遣費是幾個月薪水？」
AI 錯誤回答：「3 個月薪水」
正確答案：「3 個月 × 0.5 = 1.5 個月薪水」

用戶依據錯誤資訊與老闆談判，損失 1.5 個月薪水
```

**緩解策略（多層防護）**

```
Layer 1: Prompt 工程
    → 強制要求引用法條
    → 不確定時必須說「不確定」
    → 複雜案件主動轉介

Layer 2: RAG 強制引用
    → 沒有檢索到相關法條 → 拒絕回答
    → 相似度 < 0.7 → 標示「低信心度」

Layer 3: 信心度評分
    → 高（🟢）：有明確法條 + 問題常見
    → 中（🟡）：有法條但需推論
    → 低（🟠）：資訊有限，建議諮詢專業
    → 無法回答（🔴）：超出範圍，立即轉介

Layer 4: 人工抽查
    → 每週隨機抽查 50 筆回答
    → Legal Advisor 評分
    → 錯誤率 > 2% → 觸發緊急審查

Layer 5: 用戶回報
    → 🚨 回報錯誤 按鈕
    → 4 小時內人工審核
    → 24 小時內修正

Layer 6: 免責聲明
    → 首次使用：完整免責聲明（必須閱讀）
    → 每次回答：頁尾簡短聲明
    → 複雜案件：加強警示
```

| 41-50 | 職場問題 | 職場霸凌怎麼辦？職災怎麼申請？ | 中 |

**驗證方式**
- 分析 PTT、Dcard 勞工版關鍵字
- Google Trends「勞基法 + X」查詢量
- 1955 專線常見問題統計（如可取得）

---

#### Decision 5: 內部文章審核流程

**建議採用三級審核**（詳見 [dual-source-rag-strategy.md](#)）

```
Level 1: 作者自我審核
    → 30+ 項檢查清單
    → 預估通過率：70%
         ↓
Level 2: Content Lead 同儕審核
    → 可讀性、完整性、一致性
    → 預估通過率：85%
         ↓
Level 3: Legal Advisor 法律審核
    → 準確性、風險管理
    → 預估通過率：95%
         ↓
    發布
```

**時間估算**
- 單篇 FAQ 撰寫：4-6 小時
- Level 1 審核：1 小時
- Level 2 審核：1 小時
- Level 3 審核：0.5 小時
- 修正與調整：2 小時
- **總計**：8-10 小時/篇

**50 篇 FAQ 總時程**：約 6-8 週（1 位 Content Writer + 兼職 Legal Advisor）

### 6.3 商業模式決策（需於 Phase 1 完成前確定）

#### Decision 6: 是否收費？

**選項分析**

| 模式 | 優點 | 缺點 | 建議 |
|------|------|------|------|
| **完全免費** | • 符合使命<br>• 用戶成長快<br>• 社會影響力 | • 無收入<br>• 需外部資金 | ✅ **建議 MVP** |
| **Freemium** | • 核心免費<br>• 進階收費<br>• 可永續 | • 可能排除弱勢<br>• 違反憲章 | ⏳ Phase 3 評估 |
| **訂閱制** | • 穩定收入 | • 違反「永久免費」承諾 | ❌ 不建議 |
| **B2B 授權** | • 企業付費<br>• 不向勞工收費 | • 需企業版功能 | ⏳ Phase 3 考慮 |

**建議策略**
```
MVP - Phase 2: 完全免費
    ↓
Phase 3: 評估 B2B 模式
    • 企業 HR 版本（進階功能）
    • 政府單位授權
    • 勞工團體合作
    ↓
核心功能永久對勞工免費
```

## 9. 結論與建議

### 9.1 核心建議總結

#### ✅ 強烈建議採納

1. **雙資料來源定位明確**
   - 法條 = Primary（權威基礎）
   - 內部文章 = Secondary（實務指引）
   - 清楚視覺區分

2. **MVP 範圍務實**
   - 法規：8 大法規核心條文（~176 條）
   - 內部文章：Top 50 FAQ + 10 篇行動指南
   - 時程：28 週

3. **品質控制嚴格**
   - 法條：雙源驗證 + 自動監控
   - 內部文章：三級審核制度
   - AI 回答：信心度標示 + 強制引用

4. **技術選型穩健**
   - Vector DB: Pinecone（快速上線）
   - LLM: Claude 3.5 Sonnet（繁中強）
   - Embedding: text-embedding-3-large

---

#### ⚠️ 需要特別注意

1. **法律風險管理**
   - 加強免責聲明（首次使用 + 每次回答）
   - 複雜案件主動轉介
   - 建立錯誤回報與快速修正機制

2. **成本控制**
   - LLM API 費用監控（預估 NT$ 21K/月）
   - 實作快取機制（可省 30-50%）
   - 初期用較小規模雲端資源

3. **內容維護**
   - 法規更新 SLA 24 小時（需自動化監控）
   - 內部文章定期審核（每 6 個月）
   - 用戶回饋驅動優化

---

### 9.2 成功關鍵因素

1. **Legal Advisor 深度參與**
   - 不只審核內容，需參與產品設計
   - 確保法律風險可控
   - 建議至少 0.5 FTE（可考慮增至 1 FTE）

2. **用戶研究驗證假設**
   - MVP 前進行用戶訪談（15-20 人）
   - 驗證 FAQ 優先級
   - 測試分層呈現是否有效

3. **技術與內容並重**
   - 技術再強，內容不準確 = 失敗
   - 內容再好，技術太慢 = 無法規模化
   - 兩者需平衡投入

4. **持續優化機制**
   - 每週檢視指標
   - 每月優化低表現內容
   - 每季策略調整

### 9.4 風險提醒

🔴 **高風險（需立即處理）**

- 成本超出預算 → 快取機制可控

### 9.5 最後建議

基於對 Labor Law Assistant 專案的深入分析，我認為「雙資料來源 RAG 策略」是**正確且必要**的產品方向。這不僅是技術架構的選擇，更是**用戶價值**的體現：

- **法條資料庫**提供「法律上站得住腳」的依據
- **內部文章庫**提供「人能理解、能行動」的指引
- **兩者結合**才能真正實現「讓每一位台灣勞工都能輕鬆理解自己的勞動權益」的願景

但成功的關鍵不在技術本身，而在於：
1. **嚴格的品質控制**（三級審核、信心度標示）
2. **持續的用戶研究**（驗證假設、優化體驗）
3. **負責任的 AI 使用**（清楚限制、主動轉介）
